# Nexus Audio — รายงานข้อเสนอแนะการพัฒนาและเพิ่มประสิทธิภาพ (Roadmap)
> สถานะแอปพลิเคชัน: Stable Release (v3) ไร้บั๊ก
> เอกสารฉบับนี้: สรุปแผนงานสำหรับพัฒนาต่อยอด

เอกสารฉบับนี้จัดทำขึ้นเพื่อนำเสนอแนวทางการพัฒนาแอปพลิเคชัน Nexus Audio ในอนาคต โดยแบ่งออกเป็น "การเพิ่มประสิทธิภาพระบบเดิม" (Optimizations) และ "การสร้างระบบใหม่" (New Features) เพื่อยกระดับประสบการณ์ผู้ใช้และประสิทธิภาพการทำงานของแอปให้ก้าวไปอีกขั้น

---

## 🔵 1. ระบบเดิมที่ควรพัฒนาต่อให้มีประสิทธิภาพสูงขึ้น (Optimizations & Refactoring)

### 1.1 ปรับปรุงระบบฐานข้อมูลและสถานะ (State & Data Management)
* **ปัญหาเดิม:** ปัจจุบันแอปเก็บประวัติดาวน์โหลดและเพลย์ลิสต์ลงไฟล์ JSON ซึ่งหากผู้ใช้สะสมเพลงไว้จำนวนหลักหมื่น การโหลด/เขียนไฟล์ทั้งหมดอาจทำให้ระบบช้าและกิน RAM
* **แนวทางพัฒนา:** เปลี่ยนไปใช้ฐานข้อมูลอย่าง **SQLite (ผ่าน `better-sqlite3`)** ในฝั่ง Main Process หรือ **IndexedDB** ในฝั่ง Renderer เพื่อให้สามารถค้นหา จัดเรียง และแบ่งหน้าดึงข้อมูล (Pagination) ได้อย่างรวดเร็วโดยไม่ต้องโหลดเข้า RAM ทีเดียวทั้งหมด

### 1.2 ปรับปรุงสถาปัตยกรรมการประมวลผลเสียง (Audio Engine & EQ)
* **ปัญหาเดิม:** การใช้ `BiquadFilterNode` สร้าง Equalizer 10 แบนด์ต่อตรงเป็นทอดๆ อาจทำให้เกิด Phase Distortion หรือเสียงแตก (Clipping) หากผู้ใช้ดึง Gain ขึ้นสูงสุดทุกย่าน
* **แนวทางพัฒนา:** 
  * นำระบบ **Compressor / Limiter Node** เข้ามาต่อท้ายวงจร EQ เพื่อคุมความดังไม่ให้เกิน 0dBFS และป้องกันเสียงแตก
  * ย้ายการประมวลผลหนักๆ ของตัว Audio Visualizer ไปใช้ **AudioWorklet** หรือ **WebGL** แทน Canvas 2D ปกติ เพื่อประหยัดทรัพยากร CPU

### 1.3 ปรับปรุงโครงสร้าง UI (Component-Based Refactoring)
* **ปัญหาเดิม:** ไฟล์ `renderer.js` มีขนาดใหญ่และรวบรวมฟังก์ชันทุกอย่างไว้ในไฟล์เดียว ทำให้ดูแลรักษาและเพิ่มฟีเจอร์ในอนาคตได้ยาก
* **แนวทางพัฒนา:** ควรปรับโครงสร้างระบบให้เป็น **Component-based** โดยอาจย้ายไปใช้ Framework ขนาดเบาอย่าง Vue.js, Svelte หรือ React เพื่อแยกการทำงานของ Player, Playlist, Settings และ Downloader ออกเป็นสัดส่วน

### 1.4 ดึงข้อมูลเพลงแบบ Lazy Loading
* **ปัญหาเดิม:** การดึง Metadata แบบ Background แม้จะจำกัดคิวแล้ว (Parallel Limit) แต่ถ้านำเข้าไฟล์เยอะๆ ระบบก็ยังต้องรันคิวจนจบ
* **แนวทางพัฒนา:** ทำระบบ **Lazy Load Metadata** โดยใช้ `IntersectionObserver` ดึงหน้าปกและชื่อเพลงเฉพาะรายการที่ผู้ใช้เลื่อนหน้าจอ (Scroll) ไปเห็นเท่านั้น

## 🟢 2. ระบบใหม่ที่ควรเพิ่มเข้ามา (Proposed New Features)

### 2.1 ระบบดาวน์โหลดอัจฉริยะ (Smart Downloader)
* **รองรับการโหลดทั้งเพลย์ลิสต์ (Playlist Download):** ปัจจุบันโหลดได้ทีละคลิป ควรเพิ่มการวางลิงก์ YouTube Playlist แล้วให้ระบบสแกนดึงออกมาทั้งหมดในคลิกเดียว
* **ตั้งเวลาและซิงค์อัตโนมัติ (Auto-Sync):** ระบบ Subscribe ช่อง หากพบวิดีโอเพลงใหม่ ให้ตัวแอปดาวน์โหลดพื้นหลังมาเก็บไว้ให้โดยอัตโนมัติ

### 2.2 ระบบห้องสมุดเพลงและค้นหาขั้นสูง (Advanced Library & Global Search)
* **จัดการคลังเพลง (Library Management):** เพิ่มหน้า UI สำหรับจัดเรียงเพลงตาม ศิลปิน (Artist), อัลบั้ม (Album), หรือความยาวเพลง แทนที่จะมองเป็นคิวคิวเดียว
* **Global Search:** เพิ่มช่องค้นหาหลักที่พิมพ์ครั้งเดียว แล้วระบบจะค้นหาให้ทั้งใน เพลย์ลิสต์, ประวัติดาวน์โหลด, และไฟล์ในโฟลเดอร์

### 2.3 ระบบรองรับเนื้อเพลง (Synced Lyrics Support)
* **แสดงเนื้อเพลงแบบไหลตามเวลา (LRC):** รองรับการอ่านไฟล์ `.lrc` ควบคู่กับเพลง
* **โหลดเนื้อเพลงอัตโนมัติ:** เชื่อมโยงกับ API ภายนอก (เช่น LRCLIB) เพื่อดึงเนื้อเพลงมาแสดงบนแอปตามเพลงที่กำลังเปิดอยู่ทันทีโดยไม่ต้องหาโหลดเอง

### 2.4 ปรับปรุงการควบคุมระดับ OS (Native OS Integrations)
* **Global Hotkeys:** ให้ผู้ใช้ตั้งค่าคีย์ลัดบนคีย์บอร์ด (เช่น Ctrl+Alt+P) เพื่อสั่งเล่น/หยุด หรือเปลี่ยนเพลงได้ตลอดเวลา แม้จะพับแอปซ่อนไว้
* **Mini-Player โหมดลอยตัว (Always on Top):** เพิ่มตัวเลือกให้ Mini Player ลอยอยู่เหนือหน้าต่างทุกโปรแกรม ไม่ถูกหน้าต่างอื่นบัง

### 2.5 ปลั๊กอินและการรองรับธีม (Themes & Plugins)
* **Custom Themes:** เปิดให้ผู้ใช้แต่งสี (Accent Colors) เปลี่ยนรูปพื้นหลังแอป หรือแชร์ธีมของตัวเอง
* **Plugin System:** ระบบส่วนขยาย เพื่อให้นักพัฒนาคนอื่นสามารถเขียนสคริปต์เสริม เพื่อดึงเพลงจากเว็บไซต์อื่นๆ (นอกเหนือจาก YouTube) เข้ามายังแอปได้
